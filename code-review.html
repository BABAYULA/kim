Critical AJAX handlers have no nonce verification or capability checks class-iat-main.php ›

All AJAX handlers (handle_get_quote, handle_make_booking, handle_save_zone, etc.) currently stub out with wp_die(). When you implement them, they must verify nonces and check user capabilities first — otherwise any visitor or logged-out user can trigger admin-only actions.

✦ Recommended Pattern 
public function handle_save_zone() {
    // 1. Verify nonce
    check_ajax_referer('iat_nonce', 'nonce');

    // 2. Check capability for admin-only actions
    if (!current_user_can('manage_options')) {
        wp_send_json_error(['message' => 'Unauthorized'], 403);
    }

    // 3. Sanitize input, then do work...
    $zone_data = sanitize_text_field($_POST['zone_data'] ?? '');
    // ...
    wp_send_json_success(['message' => 'Zone saved.']);
}
Critical Weak IV generation makes AES encryption predictable class-iat-memory-security.php ›

The generate_iv() method likely derives the IV deterministically from the key. AES-CBC requires a random IV per encryption operation; reusing a fixed IV leaks information about repeated plaintexts and breaks semantic security.

✦ Recommended Fix 
// ❌ Deterministic IV — don't do this
$iv = hash('md5', $key); // always same IV for same key

// ✅ Random IV, stored alongside ciphertext
$iv = openssl_random_pseudo_bytes(16);
$encrypted = openssl_encrypt($data, 'AES-256-CBC', $enc_key, 0, $iv);
// Store as: base64(iv + encrypted)
return base64_encode($iv . $encrypted);

// When decrypting:
$raw = base64_decode($stored);
$iv  = substr($raw, 0, 16);
$enc = substr($raw, 16);
return openssl_decrypt($enc, 'AES-256-CBC', $enc_key, 0, $iv);

Warning Autoloader path mapping is fragile for nested classes class-iat-autoloader.php ›

The autoloader converts IAT_DB_Manager → includes/class-iat-db-manager.php, but IAT_DB_Manager lives in includes/database/. It happens to work because the file is explicitly required elsewhere, but any future class in a subdirectory will silently fail to autoload.

✦ Fix 
// Add subdirectory mapping
private static $dir_map = [
    'DB_Manager'    => 'database/',
    'Security'      => 'security/',
    'Zone_Detector' => 'zones/',
    'API_Rotator'   => 'geocoding/',
    'Pricing_Engine'=> '',
];

public static function autoload($class_name) {
    if (strpos($class_name, 'IAT_') !== 0) return;
    $name = substr($class_name, 4); // strip IAT_
    $subdir = self::$dir_map[$name] ?? '';
    $file = IAT_PLUGIN_DIR . 'includes/' . $subdir
          . 'class-' . strtolower(str_replace('_', '-', $name)) . '.php';
    if (file_exists($file)) require_once $file;
}
Warning wp_kses_post() used on GeoJSON data — wrong sanitizer class-iat-db-manager.php ›

GeoJSON is structured JSON, not HTML. Running it through wp_kses_post() can silently strip valid JSON content (e.g. angle brackets in strings). Use wp_json_encode(json_decode($value)) to round-trip validate it instead.

✦ Fix 
// ❌ Wrong tool for the job
'geojson' => wp_kses_post($data['geojson']),

// ✅ Validate it is actually JSON
$decoded = json_decode($data['geojson'], true);
if (json_last_error() !== JSON_ERROR_NONE) {
    return false; // reject invalid GeoJSON
}
'geojson' => wp_json_encode($decoded),
Warning get_all_regions() uses unprepared query class-iat-db-manager.php ›

The query string is safe here since it has no user input, but WordPress Coding Standards flags any $wpdb->get_results() call with a bare SQL string. More importantly, if someone extends this method with dynamic conditions later, the pattern is already unsafe. Wrap it with $wpdb->prepare() or at minimum add a // phpcs:ignore comment explaining why it's safe.

// Unguarded
$wpdb->get_results("SELECT * FROM {$this->get_regions_table()} ORDER BY region_name ASC");

// Safer pattern even for static queries
$table = esc_sql($this->get_regions_table());
$wpdb->get_results("SELECT * FROM `{$table}` ORDER BY region_name ASC"); // phpcs:ignore
Warning Booking ID generated with str_shuffle() — not cryptographically random class-iat-db-manager.php ›

str_shuffle() uses PHP's mt_rand() which is not cryptographically secure. Booking IDs and cancellation tokens are security-sensitive — a predictable ID lets someone guess/cancel another person's booking.

✦ Fix 
// ❌ Predictable
substr(str_shuffle('ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789'), 0, 8)

// ✅ Cryptographically random
'IAT' . date('Ymd') . strtoupper(bin2hex(random_bytes(4)))

Suggestion Add a database version migration system class-iat-activator.php ›

You store iat_db_version but never check it on subsequent activations. When you release v1.1.0 with schema changes, dbDelta() handles additions but not column renames or drops. Add an upgrade path now before real data exists.

public static function activate() {
    $installed_ver = get_option('iat_db_version', '0.0.0');
    if (version_compare($installed_ver, IAT_VERSION, '<')) {
        self::create_database_tables(); // dbDelta is safe to re-run
        self::run_migrations($installed_ver);
        update_option('iat_db_version', IAT_VERSION);
    }
}
